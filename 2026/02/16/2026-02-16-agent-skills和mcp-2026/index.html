Tools are the functions themselves. MCP is a standardized communication layer that helps the model discover and use those functions. Skills are high-level instruction sets that guide the model on how to orchestrate those tools effectively.

But what’s not good

先看几个概念

Integration (集成): 把两个不同的系统（比如 AI 和 天气预报网站）连在一起的过程。

  • 比如… 把你的手机连接到车载音响上，这就是一种集成。

State (状态): 指 AI 在干活过程中记得“走到哪一步了”。

  • 例如… AI 查到了机票，还没付钱，这个“已选好票但未支付”的情况就是状态。

Orchestrate (编排): 指挥多个动作按顺序完成。

  • 比如… 像指挥家一样，让 AI 先查票，再问你确认，最后扣款。

Sure 现在各种AI智能体（Agent）能操作文件、发邮件、查数据……这一切都建立在函数调用的基础上。

给模型能力很容易。只要定义一个函数 schema，模型就能学会什么时候调用。

难的是管理能力

后来 MCP、Agent 框架、Skill 系统出现的背景。

它们解决的不是：

“能不能调用工具”

而是：

“如何标准化管理一堆工具”

为什么管理能力很难

Every integration was bespoke. You had to write custom code to hook up each function, handle the results, manage state between calls, and orchestrate sequences of actions. Want your AI to use ten different APIs? Great, write ten different integrations and figure out how they compose. There was no standard, no shared infrastructure, no way to say “here’s a tool someone else built, just use it.”

And the model could handle only one function call at a time. Want to do something complex that requires coordinating multiple tools? You’re writing an orchestration layer yourself. The developer became the choreographer, manually managing every interaction between the model and the outside world.

It worked. It just didn’t scale.

为什么mcp is not enough and what is mcp

What Mcp solves is how LLM大模型 标准化tool calling

模型本身：

  • 不能访问数据库
  • 不能访问内部系统
  • 不能操作本地文件

如果你希望模型：

  • 查订单
  • 查库存
  • 发邮件
  • 运行脚本

就必须让它“调用外部程序”。

这就是 Tool Calling。

模型不拥有工具tool。用户 → 模型判断需要工具 模型输出（call）的json → 你的后端程序 → 执行真实代码=调用工具 → 返回结果 → 模型 → 用户
