        - Computed 和 Watch 的区别
          - 对于Computed：
            - 它支持缓存，只有依赖的数据发生了变化，才会重新计算
            - 不支持异步，当Computed中有异步操作时，无法监听数据的变化
            - computed的值会默认走缓存，计算属性是基于它们的响应式依赖进行缓存的，也就是基于data声明过，或者父组件传递过来的props中的数据进行计算的。
            - 如果一个属性是由其他属性计算而来的，这个属性依赖其他的属性，一般会使用computed
            - 如果computed属性的属性值是函数，那么默认使用get方法，函数的返回值就是属性的属性值；在computed中，属性有一个get方法和一个set方法，当数据发生变化时，会调用set方法。
          - 对于Watch：
            - 它不支持缓存，数据变化时，它就会触发相应的操作
            - 支持异步监听
            - 监听的函数接收两个参数，第一个参数是最新的值，第二个是变化之前的值
            - 当一个属性发生变化时，就需要执行相应的操作
            - 监听数据必须是data中声明的或者父组件传递过来的props中的数据，当发生变化时，会触发其他操作，函数有两个的参数：
              - immediate：组件加载立即触发回调函数
              - deep：深度监听，发现数据内部的变化，在复杂数据类型中使用，例如数组中的对象发生变化。需要注意的是，deep无法监听到数组和对象内部的变化。
                - 
                - 
                - 不写deep得这样才能观察
                - 
                - {deep：true}可以写对象名就可以
              - 
            - 当想要执行异步或者昂贵的操作以响应不断的变化时，就需要使用watch。

              - import{watch}

          - 总结：
            - 计算属性依赖的响应式数据和watch监听的数据都是基于data声明过，或者父组件传递过来的props中的数据
            - watch的作用更多是一个观察的监听器，对数据监听回调执行操作
            - computed很重要的是依赖和缓存，依赖于其他属性值，它自己也是一个属性
          - 运用场景：
            - 当需要进行数值计算,并且依赖于其它数据时，应该使用 computed，因为可以利用 computed 的缓存特性，避免每次获取值时都要重新计算。
            - 当需要在数据变化时执行异步或开销较大的操作时，应该使用 watch，使用 watch 选项允许执行异步操作 ( 访问一个 API )，限制执行该操作的频率，并在得到最终结果前，设置中间状态。这些都是计算属性无法做到的。
        - v-if和v-show的区别
          - 手段：v-if是动态的向DOM树内添加或者删除DOM元素；v-show是通过设置DOM元素的display样式属性控制显隐；          v-if是添加果子，v-show是更改果子是否可见，果子已经被渲染在那里
          - 编译过程：v-if切换有一个局部编译/卸载的过程，切换过程中合适地销毁和重建内部的事件监听和子组件；v-show只是简单的基于css切换；  v-if切换殃及这个果子下结的其他果子和叶子
          - 是否被编译：v-if是惰性的，如果初始条件为假，则什么也不做；只有在条件第一次变为真时才开始局部编译; v-show是在任何条件下，无论首次条件是否为真，都被编译，然后被缓存，而且DOM元素保留；
          - 性能消耗：v-if有更高的切换消耗；v-show有更高的初始渲染消耗；
          - 使用场景：v-if适合运营条件不大可能改变；v-show适合频繁切换。

          - 在实现协同编辑功能时，需要考虑以下技术关键点：
          - 1. **实时通信**：确保所有参与者之间的实时通信是关键。这可以通过 WebSocket 或者 WebRTC 等技术实现。WebSocket 提供了双向通信的能力，而 WebRTC 则可以支持更复杂的实时通信场景，比如音视频通话。
          - 2. **数据同步**：保证所有用户的编辑操作能够同步到其他用户的客户端，以确保他们在同一个文档上看到相同的内容。这可以通过实时数据库（如 Firebase Realtime Database 或者 MongoDB Change Streams）来实现，也可以使用自定义的后端服务器来处理编辑操作并将其广播给其他参与者。
          - 3. **冲突解决**：在多个用户同时编辑同一个文档时，可能会出现冲突。因此，需要实现一套冲突解决机制，以确保文档的一致性。常见的解决方案包括 OT (Operational Transformation) 和 CRDT (Conflict-free Replicated Data Types) 等算法。
          - 4. **用户身份认证与权限管理**：确保只有授权用户可以编辑文档，并且能够识别不同用户的编辑操作。这可以通过用户身份认证和权限管理系统来实现，比如使用 JSON Web Tokens (JWT) 进行用户认证，并设置不同的权限级别来控制用户对文档的操作。
          - 5. **性能优化**：实时协同编辑可能会面临大量的网络请求和数据传输，因此需要进行性能优化，包括减少数据传输量、合并编辑操作、客户端缓存等。
          - 总的来说，实现协同编辑功能需要综合运用实时通信、数据同步、冲突解决、用户身份认证与权限管理以及性能优化等多种技术和算法。

            - 为什么要有同源策略
              - 同源策略规定，只有同源的页面之间才能进行某些类型的交互，例如读取 DOM、发送 AJAX 请求等。这是为了保护用户的安全，防止恶意的第三方网站访问或修改用户的数据。
              - 例如，假设你在浏览器中打开了银行的网站，并在另一个标签页中打开了一个恶意网站。如果没有同源策略，那么恶意网站的脚本就可能能够访问银行网站的 DOM，读取你的账户信息，或者模拟你的操作，进行转账等操作。有了同源策略，这种情况就不会发生，因为浏览器会阻止来自恶意网站的脚本访问银行网站的 DOM。
            - 浏览器的同源策略是一种安全机制，它阻止了来自不同源的脚本进行交互
            - 这意味着，如果你的网页试图使用 AJAX（异步 JavaScript 和 XML）从不同的源获取数据，浏览器会阻止这种行为。

          - 虚拟 DOM
          - 虚拟 DOM 的“虚”主要体现在它并不直接对应浏览器中的实际 DOM 结构，而是在内存中维护一棵虚拟的 DOM 树。
          - 内存中的表示： 虚拟 DOM 是在 JavaScript 内存中以对象的形式存在的，它是对实际 DOM 结构的一种抽象表示。这意味着开发者可以使用 JavaScript 来操作虚拟 DOM，而不必直接操作实际的 DOM，从而减少了与浏览器交互的次数。
          - 对比和更新： 当应用程序的状态发生变化时，框架会根据新的状态生成一个新的虚拟 DOM 树。然后，框架会将新生成的虚拟 DOM 树与之前的虚拟 DOM 树进行比较，找出两者之间的差异。最后，框架会将这些差异应用到实际的 DOM 上，从而更新用户界面。
          - 与实际 DOM 的关系： 虚拟 DOM 最终会被转换成实际的 DOM，并渲染到浏览器中。但是，虚拟 DOM 通过一系列的优化手段，比如批量更新、异步更新等，使得与实际 DOM 的交互次数大大减少，从而提高了性能。

          - http状态码 HTTP 状态码用于表示服务器对请求的处理结果。它是由三位数字和一个可选的原因短语组成 1xx（信息响应）：表示接收到请求并且继续处理。 2xx（成功）：表示请求已被成功接收、理解、并接受。 3xx（重定向）：需要进一步操作以完成请求。 4xx（客户端错误）：请求包含语法错误或无法完成请求 5xx（服务器错误）：服务器在处理请求的过程中发生了错误。
        - 301与302
          - 301 Moved Permanently (永久重定向)：
            - 当服务器返回状态码 301 时，它表示请求的资源已被永久移动到了新的 URL 地址。
            - 客户端收到 301 状态码后，会自动使用新的 URL 发起新的请求，而不再使用原始的 URL。
            - 搜索引擎通常会更新其索引，将原始 URL 替换为新的 URL。
          - 302 Found (临时重定向)：
            - 当服务器返回状态码 302 时，它表示请求的资源已暂时移动到了新的 URL 地址。
            - 客户端收到 302 状态码后，会使用新的 URL 发起新的请求，但未来可能会继续使用原始的 URL。
            - 搜索引擎通常不会更新其索引，保留原始 URL。
          - 区别总结如下：
          - 301 表示永久重定向，客户端会记住新的 URL 并使用它来访问资源；302 表示临时重定向，客户端会暂时访问新的 URL，并可能在将来继续使用原始的 URL。
          - 搜索引擎通常会更新索引以反映 301 的改变，但不会更新 302 的改变。

          - 在 Vue3.0 中已经不使用这种方式了，而是通过使用 Proxy 对对象进行代理，从而实现数据劫持。使用Proxy 的好处是它可以完美的监听到任何方式的数据改变，唯一的缺点是兼容性的问题，因为 Proxy 是 ES6 的语法。

        - SPA和vue
          - 在单页应用中，当你首次访问应用时，服务器会发送一个 HTML 页面到浏览器。这个页面通常包含应用的骨架（如导航栏、页脚等）和一些脚本链接。（index.html)
          - 在页面加载后，Vue.js（或其他前端框架）会接管页面，所有的用户交互（如点击链接、提交表单等）都会被 Vue.js 捕获。然后根据应用的路由配置，决定显示哪个组件。这个过程完全在浏览器中完成，无需向服务器发送请求，也无需重新加载页面。这就是为什么它被称为"单页"应用。当用户请求新的"页面"时，不会发送新的 HTTP 请求到服务器，而是由 Vue.js 在浏览器中处理。
          - Vue.js 通过切换显示在页面上的组件来实现"页面"的切换。例如，当用户点击一个链接时，Vue.js 可能会移除当前的组件，并替换为另一个组件，从而实现了"页面"的切换，但实际上整个过程都在一个 HTML 页面中完成，没有进行页面的重新加载。
          - 这种方式的优点是用户体验流畅，因为页面切换无需等待服务器响应。但缺点是首次加载应用时，需要下载所有的 JavaScript 和 CSS，这可能会导致首屏加载时间较长。为了解决这个问题，通常会使用代码分割和懒加载技术，只在需要时才加载某些 JavaScript 和 CSS。

              - 组件应用到DOM上需要挂载                > **在 Vue.js 中，挂载将Vue 实例（或组件）与 DOM 关联起来。挂载的过程实际上是将 Vue 实例的模板转换为真实的 DOM 节点，并插入到挂载点（一个真实的 DOM 元素）中。
当你创建一个 Vue 实例时，它并不会立即影响 DOM，只有当你调用 mount 方法或者在创建实例时使用 el 选项，Vue 实例才会被挂载，也就是说，它的模板才会被转换为真实的 DOM 并插入到页面中。
这样做的好处是，你可以在挂载之前对 Vue 实例进行配置或修改，例如添加生命周期钩子、计算属性、方法等。然后，当你准备好让 Vue 实例影响 DOM 时，再进行挂载。
此外，这也使得服务器端渲染成为可能。在服务器端，你可以创建和配置 Vue 实例，然后将其渲染为字符串，而不是真实的 DOM，这样可以提高首次加载的性能。**

然而，对于基本类型（如字符串、数字和布尔值），情况就不同了。当我们将基本类型的值作为参数传递给函数或组件时，实际上传递的是这个值的副本，而不是原始值的引用。这意味着，如果函数或组件修改了这个参数，原始值不会被修改。
在 Vue.js 中，我们通常不希望子组件直接修改父组件传递下来的 props，因为这可能会导致数据流难以理解和跟踪。但是，有时我们确实需要子组件能够修改父组件的某个状态。在这种情况下，我们可以使用 Vue 3 的 ref 函数来创建一个响应式引用，然后将这个引用作为 prop 传递给子组件。这样，子组件就可以通过修改这个引用的 .value 属性来修改父组件的状态。

        - 垃圾收集

            - JavaScript具有自动垃圾回收机制，即所需内存的分配和无用内存的回收实现自动管理
            - 原理是  垃圾收集器按照固定的时间间隔去 找出不再继续使用的变量  然后释放它的内存。 像环卫工人每天早上去收集垃圾 清理街道
            - 局部变量是这样的：在函数执行过程中,程序为局部变量在堆或者栈上面开辟一个空间储存他们的值，在函数执行完毕以后就释放这部分内存
            - 什么是垃圾              > 程序不用的内存，或者是程序已经用过了，以后不会再用的内存。或者说是：不可达的值
              - 
              - 1,2就是垃圾
            - JavaScript的引用数据类型如何储存              > 保存在堆内存中的，然后在栈内存中保存一个对堆内存中实际对象的引用（即地址）      JavaScript 中对引用数据类型的操作都是操作 对象的引用而不是实际的对象。可以简单理解为，栈内存中保存了一个地址，这个地址和堆内存中的实际值是相关的
            - 为什么需要垃圾清除              > let test = {
  name: "isboyjc"
};
test = [1,2,3,4,5]
              - 
              - 没有了引用关系，也就是无用的对象，这个时候假如任由它搁置，一个两个还好，多了的话内存也会受不了，所以就需要被清理（回收）.    用官方一点的话说，程序的运行需要内存，只要程序提出要求，操作系统或者运行时就必须提供内存，那么对于持续运行的服务进程，必须要及时释放内存，否则，内存占用越来越高，轻则影响系统性能，重则就会导致进程崩溃
            - 可达性              > 以某种方式可访问的值被存储在内存中，反之不可访问则需回收
如何回收，其实就是怎样发现这些不可达的对象（垃圾）

            - 为什么不是实时的找出无用内存并释放呢？              > 开销太大了

            - 根 对象              > 引擎在执行 GC（使用标记清除算法）时，需要从出发点去遍历内存中所有的对象去打标记，而这个出发点有很多，我们称之为一组 根 对象，而所谓的根对象，其实在浏览器环境中包括又不止于 全局Window对象、文档DOM树 等
即引擎去打标记的出发点的集合

            - 为什么需要分代式？              > 垃圾收集器会为新生代分配一小块内存，然后频繁地在这块内存上进行快速清理。这种方式可以快速回收大量的短生命周期的对象，从而保持内存的使用效率。
老生代则包含存活时间长的对象，这些对象可能被程序长时间持有，或者从新生代晋升过来。老生代的垃圾收集频率较低，但每次收集时会处理更大的内存区域。
这种分代式垃圾收集机制可以根据对象的生命周期来优化内存管理，提高垃圾收集的效率。
              - 分代式机制把一些新、小、存活时间短的对象作为新生代，采用一小块内存进行频率较高的快速清理，而一些大、老、存活时间长的对象作为老生代，使其很少接受检查，新老生代的回收机制及频率是不同的，可以说此机制的出现很大程度提高了垃圾回收机制的效率

          - 分代式垃圾回收            > V8对垃圾回收的优化
我们上面所说的垃圾清理算法在每次垃圾回收时都要检查内存中所有的对象，这样的话对于一些大、老、存活时间长的对象来说同新、小、存活时间短的对象一个频率的检查很不好，因为前者需要时间长并且不需要频繁进行清理，后者恰好相反，怎么优化这点呢？？？分代式就来了

            - V8 中将堆内存分为新生代和老生代两区域，采用不同的垃圾回收器也就是不同的策略管理垃圾回收
            - 新生代的对象为存活时间较短的对象，简单来说就是新产生的对象，通常只支持 1～8M 的容量，而老生代的对象为存活事件较长或常驻内存的对象，简单来说就是经历过新生代垃圾回收后还存活下来的对象，容量通常比较大
            - 
            - 新生代垃圾回收              > 新生代的内存被分为两个半区，通常称为"From"区和"To"区。开始时，所有的对象都在"From"区，"To"区是空的。
当垃圾收集开始时，垃圾收集器会遍历"From"区的所有对象，找出那些还在活动的对象（即被其他对象引用的对象），然后将这些活动对象复制到"To"区。复制完成后，"From"区中的所有对象（无论是否活动）都被视为垃圾，并被一次性清理掉。最后，"From"区和"To"区的角色互换：原来的"To"区（现在包含所有活动对象）变成新的"From"区，原来的"From"区（现在已经空了）变成新的"To"区。这样，下一次垃圾收集时，新的活动对象会被复制到新的"To"区。
这种"复制"策略的好处是，清理垃圾时不需要逐个处理对象，只需要一次性清空整个区域，这可以大大提高垃圾收集的效率。同时，由于活动对象在复制到"To"区时会进行排序，这也有助于减少内存碎片。

这种角色翻转的操作还能让新生代中的这两块区域无限重复使用下去。

              - 新生代对象是通过一个名为 Scavenge 的算法进行垃圾回收，在 Scavenge算法 的具体实现中，主要采用了一种复制式的方法即 Cheney算法
              - Cheney算法 中将堆内存一分为二，一个是处于使用状态的空间我们暂且称之为 使用区，一个是处于闲置状态的空间我们称之为 空闲区
              - 
              - 新加入的对象都会存放到使用区，当使用区快被写满时，就需要执行一次垃圾清理操作
              - 当开始进行垃圾回收时，新生代垃圾回收器会对使用区中的活动对象做标记，标记完成之后将使用区的活动对象复制进空闲区并进行排序，随后进入垃圾清理阶段，即将非活动对象占用的空间清理掉。最后进行角色互换，把原来的使用区变成空闲区，把原来的空闲区变成使用区
              - 当一个对象经过多次复制后依然存活，它将会被认为是生命周期较长的对象，随后会被移动到老生代中，采用老生代的垃圾回收策略进行管理
              - 另外还有一种情况，如果复制一个对象到空闲区时，空闲区空间占用超过了 25%，那么这个对象会被直接晋升到老生代空间中，设置为 25% 的比例的原因是，当完成 Scavenge 回收后，空闲区将翻转成使用区，继续进行对象内存的分配，若占比过大，将会影响后续内存分配
            - 老生代垃圾回收
              - 对于大多数占用空间大、存活时间长的对象会被分配到老生代里，因为老生代中的对象通常比较大，如果再如新生代一般分区然后复制来复制去就会非常耗时，从而导致回收执行效率不高，所以老生代垃圾回收器来管理其垃圾回收执行，它的整个流程就采用的就是上文所说的标记清除算法了
              - 首先是标记阶段，从一组根元素开始，递归遍历这组根元素，遍历过程中能到达的元素称为活动对象，没有到达的元素就可以判断为非活动对象。清除阶段老生代垃圾回收器会直接将非活动对象，也就是数据清理掉。前面我们也提过，标记清除算法在清除后会产生大量不连续的内存碎片，过多的碎片会导致大对象无法分配到足够的连续内存，而 V8 中就采用了我们上文中说的标记整理算法来解决这一问题来优化空间

            - slot-scope 是一个特殊的属性，用于接收插槽的数据。在这个例子中，slot-scope="scope" 表示我们将接收到的插槽数据命名为 scope，然后我们可以在模板中使用 scope.row 来访问当前行的数据。
            - label 属性在 el-radio 组件中用于定义单选按钮的值。当单选按钮被选中时，v-model 绑定的值会更新为 label 的值。
            - scope 是我们用 slot-scope 属性接收到的插槽数据。它不是在某个地方定义的，而是由 Element UI 在渲染表格时自动提供的。scope.row 就是当前行的数据。
            - v-model 绑定 currentRow 是为了实现单选功能。当我们选中一个单选按钮时，v-model 绑定的值（即 currentRow）会更新为该单选按钮的 label 值。然后我们可以通过 currentRow 来知道哪个单选按钮被选中。
            - :label="scope.row.address" 是一个动态属性绑定，它将 label 属性绑定到当前行的 address 字段。这样每个单选按钮的 label 值都会是唯一的，我们就可以通过 label 值来确定哪个单选按钮被选中。
